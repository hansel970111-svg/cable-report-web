## Python correction to the original plan

The original pins are superseded by this security-compatible set:

```text
PyMuPDF==1.26.7
pdfplumber==0.11.10
reportlab==5.0.0
pyinstaller==6.21.0
Pillow==12.2.0
pytest==9.0.3
```

Rationale:

- PyMuPDF `1.26.5` is affected by a published advisory fixed in `1.26.7`.
- pdfplumber `0.11.10` requires Pillow `>=12.2`; the planned Pillow `11.3.0` is both incompatible and affected by advisories fixed in `12.2.0`.
- pytest `8.4.2` is affected by an advisory fixed in `9.0.3`.

`requirements.lock` and `requirements-dev.lock` are universal Python 3.12 locks generated by exactly `uv==0.11.28` with:

```text
--universal --python-version 3.12 --generate-hashes --only-binary :all:
```

This replaces the original single-host pip-tools command. The policy verifier rejects missing hashes, non-exact requirements, runtime/development version/marker/hash-set drift, missing Windows/macOS marker dependencies, an unapproved lock builder, and every requirements-file directive other than indented SHA-256 hash continuations. In particular, lock files cannot override the binary-only policy or redirect the package index.

## Node lock and vendored SheetJS policy

- Registry traffic is restricted to the official npm registry.
- Every direct dependency and development dependency is an exact version (the only non-registry exception is the approved local SheetJS tarball); `latest`, ranges, Git URLs, and HTTP package sources are rejected by tests.
- `autoInstallPeers` is disabled in both npm configuration and the lock.
- Direct exact pins are checked against both the root importer specifier and resolved version.
- `xlsx@0.20.3` is installed only from `vendor/xlsx-0.20.3.tgz`.
- The vendored file SHA-256, lock tarball path, and lock SHA-512 SRI are independently checked.
- The unused `shadcn` CLI dependency was removed.
- Exact transitive security overrides are recorded in `package.json`; full Node audit currently reports zero known vulnerabilities across production and build dependencies.

The verifier includes negative tests proving it rejects:

- a changed direct specifier;
- an unchanged specifier resolved to a different version;
- a redirected SheetJS tarball;
- a changed SheetJS integrity value;
- automatic peer installation;
- forbidden ghost root dependencies.

Windows process entrypoints are also exercised as behavior, not only inspected as text: Corepack `.cmd` execution uses the required shell path, while Electron development launches the resolved `electron.exe` directly with `shell:false`.

## Release gates

The Windows release workflow must complete all of these before packaging:

1. frozen pnpm install;
2. hash-enforced Python development-lock install and `pip check`;
3. Node and Python lock-policy verification;
4. full Node dependency audit at high severity or above;
5. lint, TypeScript, and unit checks;
6. full Python tests;
7. application and Python-worker builds;
8. desktop package-content verification.

The workflow is fixed to `windows-2025`, Node `24.14.0`, Python `3.12.13`, and pnpm `9.15.9`.

## PDF golden migration

Moving PyMuPDF from `1.26.5` to `1.26.7` changes the recorded renderer contract and PNG encoding only. Independent comparison established:

- 9 of 9 decoded golden pages are pixel-identical;
- all 6 semantic manifests are unchanged except renderer version and PNG SHA;
- all 18 PyMuPDF/Poppler review renders are visually correct.

The regenerated baselines are therefore an encoding/security migration, not an application-layout change.
